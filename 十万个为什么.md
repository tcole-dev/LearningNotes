#### 遍历时修改集合为什么报错？

在Java等各种语言中，都是不允许在遍历集合（List等）时对原集合进行修改的。若执意修改，不同语言往往会有不同的反应，python不报错但行为诡异，比如漏掉数据；java/C#运行时报错；Rust直接不允许编译。

java集合内部有一个modCount，记录修改（remove、add等）次数。在遍历集合时（增强for循环、forEach），会获取一个迭代器。

- 若遍历集合时修改了原集合，modCount改变，迭代器中的modCount没变，迭代器和集合对象的modCount不相等，下次next时，发现两者不同，就会直接报错。
- 若通过获取迭代器对象，并用该迭代器对象来删除、修改，会手动执行 `expectedModCount = modCount`。这样两者始终同步，就不会报错。

问题的根源在于**“信息不对称”**：

- **集合对象**知道结构变了，但它不知道谁在遍历它。
- **迭代器**在遍历，但它无法实时监控集合的所有变动。

引发的后果（python不报错为例）：

```python
items = [1, 2, 3, 4]
for item in items:
    if item == 1:
        items.remove(item)
print(items) # 输出 [2, 3, 4]，看起来对了？
# 但如果你删除的是 2，循环会跳过 3，因为 3 挪到了索引 1，而循环已经指向索引 2 了。
```

**正确做法：**获取集合的iterator，通过**迭代器对象的方法**修改，注意，不是获取迭代器后仍然直接修改集合对象

#### COW直接修改原数组为什么要加读锁？为什么CHM不用？

- CHM中关键的节点字段（val、next）都是final/volatile的，修改后get操作可以马上看到；且操作都是原子的，往往只涉及一次修改，比如遍历链表时，虽然同样加synchronized锁，但找到需要修改的节点后只需要给val重新赋值即可。

- 而COW修改的是数组中的值，不是volatile的，且涉及size、val的修改，并不是原子的，会严重影响一致性。
