# 锁机制

## 颗粒度分类

### 全局锁

全局锁 Flush tables with read lock (FTWRL)，会锁住整个数据库，只能读不能写，主要用于数据库的全库备份。但加上全局锁，整库都只能读，此时会产生一些负面情况：

- 若在主库备份加上全局锁，那么一段时间内，update、insert、delete都不能写入，此时会造成业务停摆。
- 若在从库备份加上全局锁，那么在备份期间，从库不能同步主库的binlog，导致主从延迟。（本质上和主库备份差不多）

所以，<font color=red>**现在一般尽量不使用全局锁，而是倾向于通过一致性进行备份 `mysqldump --single-transaction`**</font>。mysqldump是MySQL自带的备份工具，--single-transaction选项表示开启一个一致性事务，通过**`快照`**的方式进行备份。

前面学习事务时，提到事务隔离的实现方式：MVCC + 锁，MVCC就是通过记录快照的方式实现一致性，而在 mysqldump --single-transaction中，其底层本质也是通过该方式实现的。mysqldump命令执行后，会创建一个一致性快照事务，其隔离级别为 `可重复读` ，后续的select获取全库数据都是通过这个事务实现的（版本链 + ReadView）。



### 表级锁

#### 显式表锁

通过命令显式加上表级锁，分读、写。（LOCK TABLE WITH WRITE）

#### MDL 元数据锁

元数据锁，用于保护在访问表时，表的**结构**不改变，如：

```sql
-- 事务A
BEGIN;
SELECT * FROM orders WHERE id = 100;
-- 没提交

-- 事务B，会被阻塞，因为A还持有该表的MDL，此时表结构不能修改。
ALTER TABLE orders ADD COLUMN remark VARCHAR(100);
```

#### 意向锁

 意向锁是很特别的表级锁，它本身并不是用于直接保护数据，而是提醒事务加锁前判断该表中可能持有的行级锁。举例：

假设事务 A 想给某一行加行级排他锁，事务 B 想直接给整张表加共享锁或排他锁。

如果没有意向锁，数据库为了判断“这张表里有没有行已经被锁住”，可能要扫描很多行，代价很高。

所以数据库会这样做：

- 给某一行加锁前，先在**表上**加一个“意向锁”
- 用这个意向锁告诉别人：
   **这张表里有些行可能已经被我锁了**

这样别的事务只要看表级意向锁，就能快速判断能不能再加表锁。

常见有 IS（意向共享锁）、IX（意向排他锁）。

#### 自增锁

当表有自增主键时，执行INSERT语句时会加上自增锁，保证自增值分配正确。自增锁一般很少提及，不是很重要。



### 行级锁

InnoDB中的行级锁主要有以下几类：

#### 按锁的性质分类

- 共享锁：（读锁）多个持有共享锁的事务可以同时读取
- 排他锁：（写锁）事务持锁后可以进行写操作，其他事务不能再加共享锁、排他锁

#### 按锁的实现分类

注意：InnoDB中行级锁是通过对索引加锁实现的，以下的不同行级锁都是建立在此基础上的。

- 记录锁（Record Lock）

  通过主键、唯一索引命中一条记录时，通常加记录锁，只锁住一条记录，对前后间隙不加锁，是“最窄”的锁。

  ```sql
  select * from test where id = 1 for update
  ```

  加记录锁的核心是“唯一索引”（或者是主键，因为主键也是唯一的）

- 间隙锁（Gap Lock）

  间隙锁会锁住索引记录指向的一个范围（between，<，>等）。

  注意：间隙锁只锁每条记录之间的间隙，不锁记录，只防止插入、删除，不防止记录的修改

  ```sql
  select * from users
      where id between 7 and 13 
      for update;
  
  -- 假设我们要删除id为9的用户信息  
  DELETE FROM users WHERE id = 9;
  -- 此时会阻塞，无法删除
  ```

- 临键锁（Next-Key Lock）

  临键锁可以理解为完全版的行级锁，临键锁 = 间隙锁 + 记录锁。临键锁不仅锁间隙，同时也会锁住记录本身，既防止插入、删除，也防止修改。

## 行级锁的选择(RR)

- 当使用唯一索引（主键、unique索引）等价查询时，会加记录锁（临键锁退化），锁住范围最小，只有一条记录

- 使用唯一索引等值查询，但该记录不存在时，会加间隙锁。

  ```sql
  SELECT * FROM t WHERE id = 10 FOR UPDATE;
  -- 若不存在10的索引，存在的相邻索引为8，15
  -- 则会对（8，15）加上锁
  ```

- 普通索引等价查询，索引命中时，加上临键锁；索引未命中，加间隙锁（因为无法对某条记录加锁）

- 普通索引范围查询，和等价查询类似，同样对覆盖的每条记录加上临键锁，实际范围由扫描范围决定

  ```sql
  SELECT * FROM t WHERE id > 10 AND id < 20 FOR UPDATE;
  -- 扫描范围为（10，20）
  ```
  

**不同事务隔离级别下的锁：**

在Day3中提到，事务隔离的现代实现为 MVCC + 锁。不同事务隔离级别中，MVCC作用于快照读（普通查询），锁作用于锁定读、写。

- 读未提交：

  查询不加锁；写操作仅会加记录锁，防止多个事务同时修改。

- 读已提交：

  普通查询通过MVCC进行快照读

  锁定读、写操作仅对索引加记录锁，只锁记录，不锁间隙。禁用间隙锁、临键锁。

- 可重复读：

  普通查询使用MVCC，但只生成一次快照并重复使用

  锁定读、写操作，默认使用临键锁，但不同情况会不同地退化成记录锁、间隙锁。

- 串行化：

  没有普通查询，所有查询、写全部带临键锁。

## 锁升级

InnoDB中不存在真正意义上的 行锁 -->  表锁，但是在某些情况下，会对极大范围进行加行锁，此时表现出来的效果就类似于表级锁一样。

比如：不使用索引、索引失效时，存储引擎可能会扫描整张表，对前面扫过的记录全部加锁。

```sql
-- 事务A，已知b上没有索引，此时会全部扫描，同时给前面的记录全部加锁（无论是否命中记录）
select * from test where b = 2024 for update;

-- 事务B
update test set c = 121 where b = 78;
-- 此时事务B就会被阻塞，通过命令还可以看到，B被阻塞在了主键索引1上，也就是说是全表被加上了锁
```

## 死锁

死锁，是因为两个或更多事务互相等待对方手里的锁，造成永久阻塞的情况。

死锁的形成原因有很多：

- 表级锁死锁，事务A、B分别锁住了表1、2，想访问2、1

- 行级锁中，因为没有索引条件、索引失效，导致等价于表级锁，这种事务一多，就很容易死锁

- 两个事务分别拿着对方需要的行

  ```sql
  -- 事务A
  UPDATE account SET balance = balance - 100 WHERE id = 1; -- 锁住 id=1
  UPDATE account SET balance = balance + 100 WHERE id = 2; -- 想拿 id=2 的锁
  
  -- 事务B
  UPDATE account SET balance = balance - 50 WHERE id = 2;  -- 锁住 id=2
  UPDATE account SET balance = balance + 50 WHERE id = 1;  -- 想拿 id=1 的锁
  ```

  另一种情况：

  ```sql
  -- 事务A，通过非聚簇索引回表
  UPDATE account SET balance = balance - 100 WHERE key1 = 1; -- [key1,id]： [1, 10]、[1, 20]
  
  -- 事务B，通过非聚簇索引回表
  UPDATE account SET balance = balance - 100 WHERE key2 = 5; -- [key2,id]： [5, 20]、[5, 10]
  -- 虽然只有一条sql，但因为回表顺序不同，依然可能死锁。
  ```

**死锁的根本原因**：本质上是这三个条件同时出现：**`并发事务`**、**`加锁资源有交叉`**、**`获取锁的顺序不一致`**

**解决方法**：

1.  保证访问资源顺序一致

2. 尽量走索引，降低锁范围

3. 缩小事务范围，尽快提交，不要长时间持锁

4. 减少范围锁使用，改成精准命中主键

5. 增加重试机制，应该在业务代码中添加自动重试的机制

   <font color=pink>数据库中，死锁是无可避免的事，因此数据库有**`deadlock victim`**机制，不会因为死锁一直卡住，而是随机选择一个事务断开，让另一个事务先行。</font>

## 总结

在Day3中，学习了事务的一致性和一致性的实现MVCC，但是只凭借MVCC还不能完全解决一致性问题。MVCC解决的是主要是**`“读写冲突”`**；而锁机制则是解决**`“写写冲突”`**、**`强一致性场景（锁定读）`**。两者是互补关系。

1. 首先，探讨了锁的分类：按照颗粒度分，分为全局锁（备份用，但**现在有上位替代**）、表级锁、行级锁。表级锁和行级锁都可以分为读（共享锁）、写（排他锁）锁两类。两类锁的作用很明确，不再赘述。

2. 全局锁细分又有显式表锁、MDL等。<font color=orange>每次只要有事务运行，就会获取MDL，在加上行锁前，还会先加意向锁，若插入记录且有自增主键，还会加自增锁。</font>而InnoDB中，更主要的是行锁。

3. 行锁主要的有三种：记录锁、间隙锁、临键锁，其作用参靠上文，<font color=orange>在不同的索引选用、隔离级别下，会分别加不同的行级锁</font>。

4. 最后需要注意的是死锁问题，尤其是行锁的死锁，需要知道死锁的产生原因，由此还引出了如何解决死锁、优化锁的使用。
