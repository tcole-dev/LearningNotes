在秒杀场景中，会用到多种锁：乐观锁、Redis锁（分布式锁）。

## 全局唯一ID

当进行数据库分表、Redis集群等情况下，会用到`全局唯一ID`，实现全局唯一ID的主流方式：

- UUID，最简单的方式
- Redis实现全局唯一ID：Redis本身是全局共享的，所有服务器都可以读取数据，在Redis中存储一个自增的值，和时间戳拼接，实现全局ID。
- 雪花算法，最常用的全局ID算法。生成的是一个64位整数，构成：`0 | 时间戳 | 机器ID | 序列号`



## 乐观锁

常见的乐观锁有CAS，数据库Version乐观锁。本质上默认不会被修改，但修改前会检查是否被修改，比如CAS检查当前的值是否为之前读取的值，Version乐观锁会判断version字段是否变化（version字段每次修改后自增）。

超售问题：多线程场景下，若当前库存`count`为1，而多个线程同时通过了 count > 0的判断，进行数据库修改，此时可能会导致count多次减少，变成负数，这就是超售问题。

在秒杀场景中，为了避免超售，可以采用乐观锁的思想。每次要进行修改前，都判断库存是否还是之前进行判断时的值，若不是，说明已经有线程进行了修改，此时修改操作就会失败。

优化：若每次只要已经有线程修改即操作失败，这样成功率太低，实际上，我们并不关心是否有线程修改库存，只要仓库大于0即可。因此可以修改逻辑为判断库存是否大于0即可。

```java
 // 6.扣减库存
boolean success = seckillVoucherService.update()
        .setSql("stock = stock - 1") // set stock = stock - 1
        .eq("voucher_id", voucherId).gt("stock", 0) // where id = ? and stock > 0
        .update();
if (!success) {
    // 扣减失败
    return Result.fail("库存不足！");
}
```

<font color=red>这时有一个疑问，判定stock > 0再修改，这是原子操作吗？这个间隙时间里为什么不会有另一个线程插进来导致削减到负数？</font>

<font color=orange>因为数据库中，通过where条件筛选再修改，这本身就是一个原子操作，由数据库保证。</font>

## 一人一单逻辑

秒杀场景中，除了超售，还有一人一单问题，这个看起来实现很简单，每次修改前校验一下是否已经有购买记录即可，但如果一个用户同时发起多个请求，多个线程**同时通过了判定**，都到达了创建订单的位置，应该怎么办呢？

这个可以通过加锁实现，乐观锁适合修改操作，而涉及插入操作，则一般采用悲观锁，比如对用户ID字段进行加锁，每个用户只能有一个线程进行操作，避免同时通过判定的情况。

常见的悲观锁：synchronized、Lock（可重入锁等）……。但这些锁都是在同一主机才能生效，分布式场景下就会失效，这时就会引出`分布式锁`。

### 分布式锁的实现

- 数据库实现分布式锁：数据库的写操作会自带一个排他锁，可实现分布式锁，但性能太低
- Redis实现分布式锁：常用，高性能，可重入，支持Lua
- Redisson：框架锁，内部封装各种分布式锁，带有看门狗机制等，也很常用
- Zookeeper

### Redis实现分布式锁

Redis实现分布式锁的核心是SET NX，key不存在即加，表示加锁，key已存在不操作，表示加锁失败。这时有两个问题：

- 线程A加分布式锁，如果不设置过期时间，那么如果因为某种情况导致代码中断，那么分布式锁就不会被删除，造成死锁。
- 设置过期时间时，可能还未完成操作，锁就过期了，线程B立刻就可以获取锁，此时A释放锁，释放的不是A的锁，而是B的锁，如何解决这个问题？可以给分布式锁的value设置为一个随机值，只有判定这个值是自己设置的，才能进行删除（释放锁，否则不做操作，避免删除他人的锁）。

<font color=red>有这样一个极端情况，线程A判定一个锁为自己加上的（value为自己设置的随机值），但如果进行删除之前，锁过期了，而另一个线程B加上了锁，A就可能会删除B的锁，如何解决？</font>

<font color=orange>这里会引入Lua脚本，Lua是一种脚本语言，Redis允许通关接口执行Lua脚本，可以通过Lua脚本让 读取value、删除key 的过程成为一个原子操作，实现只要通过判定，就肯定会删除，也只会删除自己的锁。</font>

## 事务使用

一个秒杀场景中，会同时操作数据库、Redis，那么如何保证数据库和Redis的一致性？这里的一致性和缓存中提到的一致性问题也不一样，这里其实是避免出现根本不该存在的数据。

我们会采用Spring的事务管理，对秒杀方法加上@Transactional注解，数据库、Redis、后端操作任意一处出现问题均会进行回滚。要注意的是，事务注解的标记范围一定要覆盖锁的生效范围，防止事务未提交，但锁已经释放。
