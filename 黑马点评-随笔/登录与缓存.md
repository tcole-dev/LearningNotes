# 登录

## 四种登录校验方式

1. Cookie直接存储用户信息

   极简单的实现方式，但信息存储在浏览器中，容易被篡改/窃取，且Cookie本身能存储信息较少。

   只适合极简环境，不建议生产环境使用。

2. Cookie + Session

   Cookie只存一个 JSessionID，浏览器请求时自动带上。服务器上的Session存储数据。

   可通过销毁Session实现登出、踢人。比Cookie更加安全。

   多服务器（分布式）下，Session不共享，因此，**只适合传统单体服务**。

3. Token

   用户信息存储在JWT令牌（常见的Token令牌）中，Token经过签名、加密，避免篡改。

   **Token不依赖Cookie等，适配分布式环境**。但Token无法主动作废，Token直到过期前都一直有效，泄露时不能删除。

4. Redis + Token

   Token的定位类似Cookie，作为Redis中的键，信息被存储在Redis中。

   作为Cookie + Session的完全上位替代，性能更高、可主动销毁（删除Redis中Token对应的key）、且适配分布式环境。



## 过滤器、拦截器、监听器

之前在写登录校验时，差点直接用过滤器来写了，这里梳理一下三个`x x 器`之间的区别、关系。

1. 过滤器（Filter），Java EE标准（Servlet）。用于底层处理请求，在请求刚刚进入 Sevlet 容器时就进行拦截处理，可以修改request、response。但不知道Controller方法（Spring 层）。

   一般用于对请求的整体处理，如字符编码处理、日志记录、XSS过滤、跨域认证、**初步**权限校验。总之都是很底层的功能。

2. 拦截器（Interceptor），Spring标准。作用和过滤器很相近，但**作用时间晚于过滤器**，且只拦截到进入Controller的请求（过滤器拦截所有请求，包括静态资源）。比起过滤器，拦截器更贴近业务处理。

   一般用于登录校验、权限控制、接口限流等。

3. 监听器（Listener），Java EE标准（Servlet）。用于监听事件的生命周期（web应用全局上下文、session、request创建与销毁），不拦截请求。

   在经历了从 有状态的Cookie + Session 到 无状态的分布式架构 的变化后，监听器已经变得越来越边缘化、少用。

**三者各自的一些独特用途**

过滤器：底层包装Request。

```java
public class XssFilter implements Filter {
    public void doFilter(ServletRequest request, ServletResponse response, FilterChain chain) {
        HttpServletRequest req = (HttpServletRequest) request;
        // 包装 request，重写 getParameter()
        HttpServletRequest wrapped = new XssRequestWrapper(req);
        chain.doFilter(wrapped, response);
    }
}
```

拦截器：基于 Controller 方法做权限控制（精确到接口级别）

```java
// 接口所用注解
@Target(ElementType.METHOD)
@Retention(RetentionPolicy.RUNTIME)
public @interface RequireLogin {}
```

```java
public boolean preHandle(HttpServletRequest request, HttpServletResponse response, Object handler) {
    if (handler instanceof HandlerMethod) {
        HandlerMethod method = (HandlerMethod) handler;
        RequireLogin annotation = method.getMethodAnnotation(RequireLogin.class);
        if (annotation != null) {
            // 校验登录
        }
    }
    return true;
}
```

监听器：通过监听Session创建/销毁，统计在线人数。（只用于基于 Session 的登录体系）

```java
@WebListener
public class OnlineUserListener implements HttpSessionListener {
    private static int count = 0;
    public void sessionCreated(HttpSessionEvent se) {
        count++;
    }
    public void sessionDestroyed(HttpSessionEvent se) {
        count--;
    }
}
```

# 缓存

并发量很大时，不可能让所有请求都去查数据库，数据库肯定扛不住，所以引入了缓存，每次需要获取数据时，应该先去查一遍缓存，如果缓存中就有数据，直接返回，否则再查数据库。

而这时，会引出一系列问题。



## 如何更新缓存

根据所需一致性的要求，一般有三种更新缓存的方式：

**内存淘汰：**redis自动进行，当redis内存达到咱们设定的max-memery的时候，会自动触发淘汰机制，淘汰掉一些不重要的数据

​	这种方式用于对一致性要求极低的情况。比如几乎不会改变的数据。

**超时剔除：**当我们给redis设置了过期时间ttl之后，redis会将超时的数据进行删除，下次查询时会对缓存进行重建。

​	这种方式对一致性有一定要求，但可以容忍一定程度上的数据延迟。

**主动更新：**每次数据更新时，手动调用方法把缓存删掉，通常用于解决缓存和数据库不一致问题。

​	这种方式一致性要求最高，缓存中随时保持最新的数据。

​	<font color=orange>为什么是删除缓存？而不是修改缓存？</font>

​	<font color=skyblue>如果修改缓存，可能多次修改后，但却没人来访问，这期间都是无效写操作，浪费性能，所以每次都只删除，直到有人来访问时，从数据库查询新数据，再进行缓存重建</font>

**更多情况下，三种方式都会使用，通常会主动使用 主动更新 + 超时剔除 更新缓存，内存淘汰是Redis自动进行的。**



## 有缓存时如何更新数据

缓存中的数据来自数据库，此时，缓存和数据库两边都有数据存储，当一个请求被接收，就需要考虑，此时应该如何修改服务器上存储的数据。

有这样三种方式：

Cache Aside Pattern 人工编码方式：缓存调用者在更新完数据库后再去更新缓存，也称之为双写方案

Read/Write Through Pattern : 由系统本身完成，数据库与缓存的问题交由系统本身去处理

Write Behind Caching Pattern ：调用者只操作缓存，其他线程去异步处理数据库，实现最终一致



为了解决缓存和数据库的一致性问题，一般会采用双写模式，也就是更新数据时，更新数据库，再更新缓存（主动更新）。

<font color=orange>为什么是先修改数据库，再删除缓存？</font>

<font color=skyblue>如果先删除缓存再修改数据库，若A先删除了缓存，还没有修改数据库，此时B没有命中缓存，去数据库中查找到了旧数据并重建了缓存（旧数据），A修改数据库后不再理会缓存，导致这个旧数据还要被使用一段时间。</font>

```
A：删除缓存 --> 修改数据库为新值 -> 结束 
B：  查DB -> 查询到旧数据 -> 重建缓存（旧数据）
```

<font color=orange>先修改数据库，再删除缓存就一定稳妥吗？</font>

<font color=skyblue>不一定。举例，现在有两个线程A、B：</font>

```
（缓存已经过期 / 不存在）	>>>>>>>>>>>>时间线>>>>>>>>>>
A：			修改数据库 -> 数据库被修改为新值 -> 删除缓存
B：查询数据库 -> 获取旧数据 -> （因某种原因延时………………）->重建缓存（旧数据） 
```

<font color=skyblue>此时，虽然是先修改数据库，再删除缓存，还是有可能导致旧数据被留在缓存里，此时就用到了**`延迟双删`**，在第一次删除缓存后，隔一段时间（一两百ms），再次删除缓存。确保旧数据被删除。</font>



## 缓存失效问题

缓存失效问题：缓存穿透、缓存击穿、缓存雪崩

### 穿透

缓存穿透指的是，客户端查询根本不存在的数据，Redis和数据库中都没有对应的记录，于是，每次这些请求都能直接穿透缓存，直接打向数据库，导致数据库被压垮，类似Dos。

```
第一次请求：			请求 -> 缓存 (没有数据) -> DB (没有数据) -> 缓存无法建立
第二、三、……次请求：    请求 -> 缓存 (没有数据) -> DB (没有数据) -> 缓存无法建立
```

常见场景：查 `id=-1`、非法乱输参数、恶意攻击

**解决方案**：

1. 缓存空对象：大量请求查询不存在的数据，直接在Redis里面，以要查询的不存在的数据为key，存一个空对象，这样这些请求就只能打到缓存上，不至于把DB打崩。

   <font color=orange>可对空对象加TTL，避免后面数据库加入这个数据后，请求还在获取空对象。</font>

2. 布隆过滤器：采用哈希思想，通过一个庞大的二进制数组，判断某个数据是否存在，若存在则放行，不存在直接拦截下来。这种方式优点在于节约内存空间，存在误判（布隆过滤器走的是哈希思想，只要哈希思想，就可能存在哈希冲突）



### 击穿

缓存击穿指的是一个热点key过期了，导致一时间大量请求突然穿透了缓存层，直接涌向数据库，把数据库打崩了。

**解决方案**：

1. 互斥锁：

   当某个key过期后，给查询数据库进行缓存重建的代码块加上锁，让同一时间只能有一个线程能够去数据库中查询这条数据。

   在单体服务下，可直接用synchronized、Lock等实现，在分布式应用中，则需要使用分布式锁。

2. 逻辑过期：

   我们可以在Redis存储的值中加入一个字段，表示其有效期，这个有效期应该小于TTL，当请求查到这个有效期已经过了后，就派出一个线程来重建缓存。<font color=orange>（可使用互斥锁，选择一个请求线程来重建。**还可以使用线程池来异步重建（推荐）**）</font>

   在此期间，由于缓存本身还没真正过期（TTL还没到0），进入缓存的请求还可以获取缓存中的值，等到缓存重建完成后，无感获取新值，通过这种方式，可以让key在过期前就被重建，避免互斥锁方案会有的阻塞。



### 雪崩

缓存雪崩是指同一时间内大量key集体过期，或是Redis宕机，导致大量请求达到数据库。

**解决方案**：

1. 给不同的Key的TTL添加随机值
2. 利用Redis集群提高服务的可用性
3. 给缓存业务添加降级限流策略
4. 给业务添加多级缓存
