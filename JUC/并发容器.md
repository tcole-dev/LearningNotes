# ConcurrentHashMap

ConcurrentHashMap是一个`线程安全的HashMap`，在性能上相较HashTable有较大的提升。ConcurrentHashMap主要有两种实现：jdk1.7和jdk1.8及以后。

## JDK1.7的实现

### 实现原理

jdk1.7中，ConcurrentHashMap的实现依赖ReentrantLock。主要部分为Segment[]、HashEntry[]。实现较简单。

首先，若要实现线程安全，第一时间最容易想到的肯定是使用synchronized，让每一时刻只有一个线程访问HashMap，HashTable就是按照这个思想来实现的，但同时若有大量线程需要put，就会造成拥塞，多个线程需要排队等候，效率极低。

JDK1.7中实现ConcurrentHashMap的原理，就是将HashMap的主数组划分为多个Segment（继承ReentrantLock），每个Segment中独立维护一个HashEntry[]，实质上就是在HashMap中维护了多个HashMap，将数据划分为多段，每段数据分别上锁，一个线程访问其中一段时，其他线程也可以访问其他数据段，实现并发。

当然，若有多个线程同时访问同一个Segment，依然会阻塞（仅限put，get不会阻塞）

![](C:\Users\t'c\Desktop\Markdowns\img\jdk1.7ConcurrentHashMap.png)

Segment[]的长度由`concurrencyLevel`（并行级别，并发数，Segment数）规定，默认为16，可手动设置，设置后不可修改，后续扩容由Segment独立对其中维护的HashEntry[]完成。

### 主要方法

- get 方法

  1. 计算 key 的 hash 值

  2. 通过 hash 值，定位到对应的 Segment 对象

  3. 再次通过 hash 值，定位到 Segment 当中数组的具体位置（某一个HashEntry）。

- put 方法

  1. 计算 key 的 hash 值

  2. 通过 hash 值，定位到对应的 Segment 对象 s

  3. 插入新值到槽 s 中
     - 获取可重入锁
     - 再次通过 hash 值，定位到 Segment 当中数组的具体位置
     - 插入或覆盖 HashEntry 对象

  7. 释放锁

## 从JDK1.7到1.8的演进

1. 1.7版本中，HashEntry类似于1.8中的Node对象，但1.8中没有Segment结构，直接用完整的Node数组存储
2. 1.7版本没有协助扩容机制（helpTransfer），因为在jdk1.7中，同一时间，一个Segment只有一个线程能操作（持锁）
3. 1.7版本中，没有树化的概念，每个HashEntry下始终是链表，不会转化为红黑树。
4. 1.8的扩容中，会将原本桶中的链表直接按条件拆分为两条链，而1.7中虽然`原理同样可用`，但实际上是`直接对每个节点重新计算index`，迁移到新index对应的HashEntry下
5. JDK1.8舍弃分段锁，在并发中，锁住的是某一个桶的头节点，并发数为Node.length，且1.8中Node[]是可以扩容的，由此可以达到极高的并发程度

由于JDK7已经几乎不再使用，该版本ConcurrentHashMap的具体实现不做详细探究

## JDK1.8的实现

JDK1.8中，ConcurrentHashMap的实现中，用CAS、synchronized关键字取代了原来的分段锁，因为不再有并发数的限制，性能有较大的提高。

### 主要数据结构

####  1. Node

普通节点：

```java
static class Node<K,V> implements Map.Entry<K,V> {
    final int hash;
    final K key;
    volatile V val;
    volatile Node<K,V> next;
}
```

说明：

- `hash`、`key` 基本不变

  **`hash` 的取值可以分成几类**：

  | hash 值         | 对应节点类型         | 含义 / 状态                                                 |
  | --------------- | -------------------- | ----------------------------------------------------------- |
  | `>=0`           | 普通 Node / TreeNode | 普通 key-value 节点或树节点，hash = key.hashCode() 的扰动值 |
  | `MOVED (-1)`    | ForwardingNode       | table扩容中，该桶已经迁移，访问时需要转发到新 table         |
  | `TREEBIN (-2)`  | TreeBin              | 该桶已经树化（红黑树），table[i] 指向 TreeBin 节点          |
  | `RESERVED (-3)` | ReservationNode      | 占位节点，表示操作正在进行（如 putIfAbsent 初始化）         |

- `val` 是 `volatile`，保证更新可见
- `next` 是 `volatile`，保证链表结构可见

#### 2. TreeNode 和 TreeBin

当某个桶里的链表过长时，会转成红黑树。

- `TreeNode` 是树节点
- `TreeBin` 是桶头包装节点，用于管理整棵树

#### 3. ForwardingNode

扩容时使用的特殊节点：

- `hash = MOVED = -1`
- 表示当前桶已经迁移到新表
- 后续线程看到它，就知道应该去新表找或者帮助扩容

#### 4. sizeCtl

`sizeCtl` 是状态控制变量，既控制初始化，也控制扩容。

它的典型含义如下：

- `0`：未初始化
- `> 0`：下一次扩容阈值
- `-1`：正在初始化
- `< 0` 的其他值：正在扩容

#### 5. table

table是ConcurrentHashMap中维护的主数组，根据，**`table.length`必须为2的幂次方**（每次扩容扩大为原来的两倍），**目的是用位运算代替取模运算，提高速度**

**index = (table.length - 1) & hash**  等价于 **index = hash % (table.length)**

根据sizeCtl扩容阈值，判断table是否需要扩容。

### 主要方法

#### 1. initTable

```java
private final Node<K,V>[] initTable() {
    Node<K,V>[] tab; int sc;
    // 验证table是否为空，是空才进行判断，尝试初始化
    while ((tab = table) == null || tab.length == 0) {
        
        // 若 sizeCtl < 0，说明已有线程进行初始化，释放时间片，暂停线程，等待其他线程初始化完成。其他线程初始化后，可在循环中判断并直接退回
        if ((sc = sizeCtl) < 0)
            Thread.yield(); // lost initialization race; just spin
        
        // CAS修改sizeCtl为-1，表示自己开始初始化
        else if (U.compareAndSetInt(this, SIZECTL, sc, -1)) {
            try {
                if ((tab = table) == null || tab.length == 0) {
                    
                    // 根据sc确定数组长度，注意！sc可能>0，因为可以在ConcurrentHashMap的有参构造方法中初始化sizeCtl的值
                    int n = (sc > 0) ? sc : DEFAULT_CAPACITY;
                    @SuppressWarnings("unchecked")
                    Node<K,V>[] nt = (Node<K,V>[])new Node<?,?>[n];
                    table = tab = nt;
                    
                    // 计算sizeCtl，等价于sizeCtl = length - (length / 4)
                    sc = n - (n >>> 2);
                }
            } finally {
                sizeCtl = sc;
            }
            break;
        }
    }
    return tab;
}
```

通过CAS，确保只有一个线程可以修改sizeCtl，并进行初始化。

#### 2. helpTransfer

```java
final Node<K,V>[] helpTransfer(Node<K,V>[] tab, Node<K,V> f) {
    Node<K,V>[] nextTab; int sc;
    
    // 确定table不为空，且当前桶已迁移（桶节点为ForwardingNode），表示可协助扩容
    if (tab != null && (f instanceof ForwardingNode) &&
        // 获取扩容后的新table
        (nextTab = ((ForwardingNode<K,V>)f).nextTable) != null) {
        
        // 获取扩容标识，扩容标识由table.length唯一确定，同时左移RESIZE_STAMP_SHIFT，左16位为扩容标识，右16位保留为扩容线程数
        int rs = resizeStamp(tab.length) << RESIZE_STAMP_SHIFT;
        
        // table、nextTable不变，且sizeCtl<0（正在扩容）时，持续判断
        while (nextTab == nextTable && table == tab &&
               (sc = sizeCtl) < 0) {
            
            // 若扩容线程达最大、只有最后一个线程在收尾、没有可领取的桶。三种情况下直接退出
            if (sc == rs + MAX_RESIZERS || sc == rs + 1 ||
                transferIndex <= 0)
                break;
            
            // CAS修改sizeCtl，将sizeCtl自增，表示扩容线程加一。因为CAS可能失败，所以使用while进行自旋
            if (U.compareAndSetInt(this, SIZECTL, sc, sc + 1)) {
                transfer(tab, nextTab);
                break;
            }
        }
        return nextTab;
    }
    
    // 不符合条件返回旧table
    return table;
}
```

扩容标识：由扩容后的table长度唯一决定，因此每一次扩容都是不同的。int类型有四个字节，左移RESIZE_STAMP_SHIFT位可将扩容标识移动到左16位，右边用于统计同时协助扩容的线程数量。**第一个线程进行扩容时，会将右16位初始化为2**，每个线程加入时加一，退出扩容时减一，若所有线程结束扩容（不包括收尾检查），sc（sizeCtl） == rs（扩容标识）+ 1

#### 4. putVal

```java
final V putVal(K key, V value, boolean onlyIfAbsent) {
    // 1. 参数校验：key和value都不能为null
    if (key == null || value == null) throw new NullPointerException();
    
    // 2. 计算hash：使用spread方法进行扰动处理
    int hash = spread(key.hashCode());
    
    // 3. binCount用于记录当前桶中的节点数量，用于后续判断是否需要树化
    int binCount = 0;
    
    // 4. 自旋循环：处理并发情况，失败自动重试
    for (Node<K,V>[] tab = table;;) {
        Node<K,V> f; int n, i, fh; K fk; V fv;
        
        // === 判断分支1：table未初始化 ===
        if (tab == null || (n = tab.length) == 0)
            tab = initTable();
        
        // === 判断分支2：目标桶为空（最优路径：无锁CAS插入）===
        else if ((f = tabAt(tab, i = (n - 1) & hash)) == null) {
            // 使用CAS尝试原子性地将新节点插入空桶
            if (casTabAt(tab, i, null, new Node<K,V>(hash, key, value)))
                break;  // CAS成功，跳出循环
        }
        
        // === 判断分支3：桶正在迁移（帮助扩容）===
        else if ((fh = f.hash) == MOVED)
            tab = helpTransfer(tab, f);
        
        // === 判断分支4：onlyIfAbsent优化（putIfAbsent且key已存在）===
        else if (onlyIfAbsent
                 && fh == hash
                 && ((fk = f.key) == key || (fk != null && key.equals(fk)))
                 && (fv = f.val) != null)
            return fv;  // 直接返回已存在的值，不插入
        
        // === 判断分支5：需要加锁处理（链表或红黑树）===
        else {
            V oldVal = null;
            
            // 获取桶头节点的锁（细粒度锁）
            synchronized (f) {
                // 双重检查：确认头节点未被修改
                if (tabAt(tab, i) == f) {
                    
                    // === 分支5.1：链表结构（fh >= 0）===
                    if (fh >= 0) {
                        binCount = 1;
                        // 遍历链表，查找或插入
                        for (Node<K,V> e = f;; ++binCount) {
                            K ek;
                            // 找到相同key
                            if (e.hash == hash &&
                                ((ek = e.key) == key ||
                                 (ek != null && key.equals(ek)))) {
                                oldVal = e.val;
                                if (!onlyIfAbsent)
                                    e.val = value;  // 覆盖旧值
                                break;
                            }
                            // 记录前驱节点，继续遍历
                            Node<K,V> pred = e;
                            if ((e = e.next) == null) {
                                // 到达链表尾部，插入新节点（尾插法）
                                pred.next = new Node<K,V>(hash, key, value);
                                break;
                            }
                        }
                    }
                    
                    // === 分支5.2：红黑树结构（TreeBin）===
                    else if (f instanceof TreeBin) {
                        Node<K,V> p;
                        binCount = 2;  // 标记为树化结构
                        // 调用TreeBin的插入方法
                        if ((p = ((TreeBin<K,V>)f).putTreeVal(hash, key,
                                                       value)) != null) {
                            oldVal = p.val;
                            if (!onlyIfAbsent)
                                p.val = value;
                        }
                    }
                    
                    // === 分支5.3：ReservationNode（computeIfAbsent专用）===
                    else if (f instanceof ReservationNode)
                        throw new IllegalStateException("Recursive update");
                }
            }
            
            // 处理树化和返回值
            if (binCount != 0) {
                // 链表长度达到树化阈值（8个节点）
                if (binCount >= TREEIFY_THRESHOLD)
                    treeifyBin(tab, i);
                // 如果是更新操作，返回旧值
                if (oldVal != null)
                    return oldVal;
                break;
            }
        }
    }
    
    // 6. 计数增加和可能触发的扩容
    addCount(1L, binCount);
    return null;
}
```

1. 参数校验（null值）计算key对应的hash值，初始化变量binCount（记录当前桶达到节点数，判断是否需要树化）

hash值的计算不是简单调用hashCode即可，spread方法可以把高位信息混入低位，减少桶分布不均的问题。

2. 自旋循环（连续判断）

   2.1 判断是否初始化（table为null或table.length==0），调用`initTable`

   2.2 若为空桶，尝试CAS插人（只有一个操作，可用CAS保证原子性）

   2.3 若该桶已经迁移，即f.hash == MOVED（table正在扩容），调用`helpTransfer方法`协助进行扩容迁移，完成后再进行put

   2.4 若是条件put，且key存在，直接返回key对应的val。（onlyIfAbsent表示只有key不存在才存放）

   2.5 若该节点下为链表/红黑树，需要深入时，使用synchronized加锁

   ​	2.5.1 若节点hash值>0，表示为普通节点，遍历链表：

   ​		找到相同key，根据onlyIfAbsent的值决定是否覆盖旧值；	若遍历完成后仍未找到key，则在链表结尾插入节点。

   ​	2.5.2 若节点是TreeBin，表示这个桶下面是红黑树结构，调用TreeBin的方法获取节点并赋值（onlyIfAbsent决定）

   ​	2.5.3 若binCount != 0，即桶中存放链表，判断是否需要树化。

3. 调用**`addCount方法`**，统计主数组中的节点数，判断是否扩容

#### 5. get

```java
public V get(Object key) {
    Node<K,V>[] tab; Node<K,V> e, p; int n, eh; K ek;
    // 获取key对应的hash值
    int h = spread(key.hashCode());
    
    // table不为空、计算得到的index对应的桶不为空时才尝试获取
    if ((tab = table) != null && (n = tab.length) > 0 &&
        (e = tabAt(tab, (n - 1) & h)) != null) {
        
        if ((eh = e.hash) == h) {
            // 1. （桶节点即要找的节点）hash值相等、key相同，返回key对应的值
            if ((ek = e.key) == key || (ek != null && key.equals(ek)))
                return e.val;
        }
        
        // 2. （特殊节点：红黑树节点、已迁移到新table）节点hash值小于0，表示为特殊节点（ForwardingNode、TreeBin）
        else if (eh < 0)
            // 统一入口方法find，获取节点，若找到则返回对应的值，若没有则返回null
            return (p = e.find(h, key)) != null ? p.val : null;
        
        // 3. （链表中的普通节点）遍历链表，若hash值相同、key相同，返回对应的值
        while ((e = e.next) != null) {
            if (e.hash == h &&
                ((ek = e.key) == key || (ek != null && key.equals(ek))))
                return e.val;
        }
    }
    // 前面的获取全部失败，返回null
    return null;
}
```

get方法不需要加锁的原因：

1. get方法不需要修改ConcurrentHashMap的结构；
2. volatile可以保证可见性，获取put后最新的值；
3. ForwardingNode可以让get方法从旧table跳转到新table获取值。

fing方法：find是不同查找的统一调用，若节点为TreeBin（红黑树），则按照红黑树的规则进行查找；若节点为ForwardingNode（已迁移），则跳转到新table后重新查找。

# CopyOnWriteArrayList

## 简述

CopyOnWriteArrayList（以下简称COW）是一种线程安全的List，其核心思想即Copy-On-Write 写时复制，每次修改时，会复制一个新的数组，修改后会用该数组直接替换原数组。

COW的线程安全主要由以下几个方面实现：

- 修改（写）时使用synchronized加锁，每次只有一个线程能进行写操作
- Copy-On-Write + volatile，COW中数组是volatile修饰的，每次写会创建一个新数组，在其中进行操作，再用新数组直接覆盖，每个线程都能直接看到最新的数组对象
- 原子操作，用新数组替换旧数组的操作是原子性的。

## 新增 add

```java
public boolean add(E e) {
    // 修改（写）时，直接加锁，确保只有一个线程可以更改数组
    synchronized (lock) {
        // 获取当前数组、长度
        Object[] es = getArray();
        int len = es.length;
        // 保留len + 1位，用于add
        es = Arrays.copyOf(es, len + 1);
        es[len] = e;
        // 覆盖原数组（原子性操作）
        setArray(es);
        return true;
    }
}
```

COW中有多种get接口，这里选择其中一种，但基本都大同小异，都遵循 **加锁 >> 获取并拷贝数组 >> 修改拷贝 >> 替代原数组** 的步骤。

<font color='red'>提问：反正都已经`synchronized`加锁了，为什么不直接操作原数组？</font>

1. <font color='skyblue'> 因为`COW`中`volatile`修饰的是数组，而不是其中的元素，若涉及其中的多个修改，根本无法触发可见性，只有直接修改原数组变量的内存地址（即重新赋值）才行。</font>
2. <font color='skyblue'>在新创建的数组中操作，不会因可见性的缺失而对旧数组产生影响，尤其是正在读的线程。</font>

## 删除 remove

```java
public E remove(int index) {
    // 写操作，先获取锁
    synchronized (lock) {
        // 获取数组
        Object[] es = getArray();
        int len = es.length;
        // 获取旧值
        E oldValue = elementAt(es, index);
        // 计算删除后要移动的元素个数（删除元素之后的元素）
        int numMoved = len - index - 1;
        Object[] newElements;
        // 删除元素为0，表示删除最后一个元素，则拷贝除最后一个元素前的所有元素
        if (numMoved == 0)
            newElements = Arrays.copyOf(es, len - 1);
        // 否则两次复制删除元素前后的两段数组
        else {
            newElements = new Object[len - 1];
            // System.arraycopy  直接调用底层C/C++库进行内存层面的拷贝，速度快
            System.arraycopy(es, 0, newElements, 0, index);
            System.arraycopy(es, index + 1, newElements, index,
                             numMoved);
        }
        // 修改原数组
        setArray(newElements);
        return oldValue;
    }
}
```

void java.lang.System.arraycopy(Object src, int srcPos, Object dest, int destPos, int length)   数组复制方法

src 要复制的对象	srcPos 开始复制的索引	dest 要复制去的对象	destPos 要复制到的索引	length 复制的长度

## COW的迭代器

COW的迭代器是基于快照实现的，每次产生迭代器、增强for循环、forEach时，都会记录一个快照，迭代器的迭代只按照这个快照来，即使原数组被修改，也不会抛出`ConcurrentModificationException`异常。

```java
// ArrayList：modCount 机制，快速失败
ArrayList<Integer> list = new ArrayList<>();
Iterator<Integer> it = list.iterator();
list.add(1);  // 修改 modCount
it.next();    // 抛出 ConcurrentModificationException

// CopyOnWriteArrayList：快照机制，安全迭代
CopyOnWriteArrayList<Integer> cowList = new CopyOnWriteArrayList<>();
Iterator<Integer> cowIt = cowList.iterator();
cowList.add(1);  // 创建新数组
cowIt.next();    // 正常，迭代的是旧快照
```

**但是！COW并不像普通集合中可以调用iterator的方法，而是：**

**<font color='red'>iterator.remove/set/add 全部抛出 UnsupportedOperationException</font>**

这里iterator、list两者的倒反天罡主要是因为一个`语义问题`。以`remove`为例子

```java
iterator.remove()
```

`iterator.remove()`规定**必须删除本次迭代中`next()`返回的对象**，但由于迭代器迭代的是旧数组（快照），修改快照没有意义，也不合语义。而新数组可能早已被修改，无法保证一定能删除next()返回的对象，无法满足语义，所以被JDK直接禁止了。

```java
list.remove()
```

`list.remove()`的语义是**删除当前array中的元素**，不需要保证删除的元素是谁，即使迭代器看到的和array删除的不同也无所谓。

就是因为这种构思💩的语义、定义，两种其实都不完全安全的方法，一个可以使用，一个不能使用。但是，因为COW用于多读少写的场景，问题不大。

# BlockingQueue

BlockingQueue（阻塞队列）与普通队列的主要区别，除了线程安全以外，还体现在Blocking 阻塞上，BLockingQueue中提供了可阻塞的api：`put`/`take`，队列中有空位，即可放入元素，若没有空位则等待有元素取出留出空位；队列中有元素，可直接获取，没有则阻塞等待放入元素。所以，堵塞队列天生适合`生产者-消费者`模型。

## ArrayBlockingQueue

ArrayBlockingQueue（以下简写ABQ），底层由一个数组存储数据，值得注意的是，ABQ是一个**循环队列**。同时，ABQ的线程安全由ReentrantLock实现。

### 主要结构/变量

```java
// lock 全局锁，每个线程操作ABQ都需要获取lock		⭐⭐⭐
final ReentrantLock lock;

// Condition条件队列，记录等待take、put的线程	  ⭐⭐⭐
private final Condition notFull;
private final Condition notEmpty;

// 统计当前节点数量
int count;									  ⭐

// 记录下一次put、take的位置，对应尾、头节点        ⭐⭐
int putIndex;
int takeIndex;

// 循环数组
final Object[] items;						   ⭐⭐

// Itrs 是一个专门用于管理所有活跃迭代器的结构。创建迭代器时，这个迭代器会被注册到 itrs 中。 ⭐
transient Itrs itrs;
```

ABQ最最核心的就是**一把锁、两个队列**。ABQ的线程安全就是由这三个变量实现的。ABQ中，无论是读或写操作，都必须获得这一把锁。当竞争失败、队列中没有足够条件（无空位或无元素）时，会分别进入两个条件队列中等待，并在满足条件时（有元素取出或有元素存入）时被唤醒。

### 主要方法

- 插入方法：
  1. add(E e) : 添加成功返回true，失败抛IllegalStateException异常
  2. offer(E e) : 成功返回 true，如果此队列已满，则返回 false。
  3. put(E e) :将元素插入此队列的尾部，如果该队列已满，则一直阻塞

- 删除方法:
  1. remove(Object o) :移除指定元素,成功返回true，失败返回false
  2. poll() : 获取并移除此队列的头元素，若队列为空，则返回 null
  3. take()：获取并移除此队列头元素，若没有元素则一直阻塞。

- 检查方法
  1. element() ：获取但不移除此队列的头元素，没有元素则抛异常
  2. peek() :获取但不移除此队列的头；若队列为空，则返回 null。

- 底层入队/出队
  - enqueue(E e)：入队方法，将节点入队到指定位置putIndex
  - dequeue()：出队方法，将指定位置takeIndex的节点出队

#### 插入

入队enqueue

```java
//入队操作
private void enqueue(E x) {
    //获取当前数组
    final Object[] items = this.items;
    //通过putIndex索引对数组进行赋值
    items[putIndex] = x;
    //索引自增，如果已是最后一个位置，重新设置 putIndex = 0;
    if (++putIndex == items.length)
        putIndex = 0;
    count++;//队列中元素数量加1
    //唤醒调用take()方法的线程，执行元素获取操作。
    notEmpty.signal();
}
```

```java
public boolean offer(E e) {
     checkNotNull(e);//检查元素是否为null
     final ReentrantLock lock = this.lock;
     lock.lock();//加锁
     try {
         if (count == items.length)//判断队列是否满
             return false;
         else {
             enqueue(e);//添加元素到队列
             return true;
         }
     } finally {
         lock.unlock();
     }
 }
//put方法，阻塞时可中断						（阻塞插入）
 public void put(E e) throws InterruptedException {
     checkNotNull(e);
      final ReentrantLock lock = this.lock;
      lock.lockInterruptibly();//该方法可中断
      try {
          //当队列元素个数与数组长度相等时，无法添加元素
          while (count == items.length)
              //将当前调用线程挂起，添加到notFull条件队列中等待唤醒
              notFull.await();
          enqueue(e);//如果队列没有满直接添加。。
      } finally {
          lock.unlock();
      }
  }
```

#### 删除/出队

出队dequeue

```java
//删除队列头元素并返回
private E dequeue() {
     //拿到当前数组的数据
     final Object[] items = this.items;
      @SuppressWarnings("unchecked")
      //获取要删除的对象
      E x = (E) items[takeIndex];
      将数组中takeIndex索引位置设置为null
      items[takeIndex] = null;
      //takeIndex索引加1并判断是否与数组长度相等，
      //如果相等说明已到尽头，恢复为0
      if (++takeIndex == items.length)
          takeIndex = 0;
      count--;//队列个数减1
      if (itrs != null)
          itrs.elementDequeued();//同时更新迭代器中的元素数据
      //删除了元素说明队列有空位，唤醒notFull条件对象添加线程，执行添加操作
      notFull.signal();
      return x;
}
```

```java
public boolean remove(Object o) {
    if (o == null) return false;
    //获取数组数据
    final Object[] items = this.items;
    final ReentrantLock lock = this.lock;
    lock.lock();//加锁
    try {
        //如果此时队列不为null，这里是为了防止并发情况
        if (count > 0) {
            //获取下一个要添加元素时的索引
            final int putIndex = this.putIndex;
            //获取当前要被删除元素的索引
            int i = takeIndex;
            //执行循环查找要删除的元素
            do {
                //找到要删除的元素
                if (o.equals(items[i])) {
                    removeAt(i);//执行删除
                    return true;//删除成功返回true
                }
                //当前删除索引执行加1后判断是否与数组长度相等
                //若为true，说明索引已到数组尽头，将i设置为0
                if (++i == items.length)
                    i = 0; 
            } while (i != putIndex);//继承查找
        }
        return false;
    } finally {
        lock.unlock();
    }
}

//根据索引删除元素，实际上是把删除索引之后的元素往前移动一个位置
void removeAt(final int removeIndex) {

     final Object[] items = this.items;
      //先判断要删除的元素是否为当前队列头元素
      if (removeIndex == takeIndex) {
          //如果是直接删除
          items[takeIndex] = null;
          //当前队列头元素加1并判断是否与数组长度相等，若为true设置为0
          if (++takeIndex == items.length)
              takeIndex = 0;
          count--;//队列元素减1
          if (itrs != null)
              itrs.elementDequeued();//更新迭代器中的数据
      } else {
      //如果要删除的元素不在队列头部，
      //那么只需循环迭代把删除元素后面的所有元素往前移动一个位置
          //获取下一个要被添加的元素的索引，作为循环判断结束条件
          final int putIndex = this.putIndex;
          //执行循环
          for (int i = removeIndex;;) {
              //获取要删除节点索引的下一个索引
              int next = i + 1;
              //判断是否已为数组长度，如果是从数组头部（索引为0）开始找
              if (next == items.length)
                  next = 0;
               //如果查找的索引不等于要添加元素的索引，说明元素可以再移动
              if (next != putIndex) {
                  items[i] = items[next];//把后一个元素前移覆盖要删除的元
                  i = next;
              } else {
              //在removeIndex索引之后的元素都往前移动完毕后清空最后一个元素
                  items[i] = null;
                  this.putIndex = i;
                  break;//结束循环
              }
          }
          count--;//队列元素减1
          if (itrs != null)
              itrs.removedAt(removeIndex);//更新迭代器数据
      }
      notFull.signal();//唤醒添加线程
}

public E poll() {
      final ReentrantLock lock = this.lock;
       lock.lock();
       try {
           //判断队列是否为null，不为null执行dequeue()方法，否则返回null
           return (count == 0) ? null : dequeue();
       } finally {
           lock.unlock();
       }
}

//从队列头部删除，队列没有元素就阻塞，可中断
 public E take() throws InterruptedException {
    final ReentrantLock lock = this.lock;
      lock.lockInterruptibly();//中断
      try {
          //如果队列没有元素
          while (count == 0)
              //执行阻塞操作
              notEmpty.await();
          return dequeue();//如果队列有元素执行删除操作
      } finally {
          lock.unlock();
      }
}
```

<font color='red'>在删除元素、出队的方法中，常常能看到一个if判断：</font>

```java
if (itrs != null)
          itrs.elementDequeued();//同时更新迭代器中的元素数据
```

在ArrayBlockingQueue中，也是支持像CopyWriteArrayList一样在迭代的同时进行修改的，但同时又会出现迭代时修改时常会出现的索引越位等问题（不会检查modCount，导致抛出异常）。前面提到`itrs`是管理迭代器的变量，每产生一个迭代器，都会在itrs中注册，当迭代时进行了删除、出队操作，`itrs.elementDequeued();`会更新迭代器中的cursor（即“同时更新迭代器中的元素数据”），防止越位等问题

<font color=orange>那么，为什么入队不需要呢？</font>

因为入队只会在队头插入数据，并不会导致迭代器的`cursor`和队列中的数据错位的情况，最多就是看不到新入队的元素。而出队、删除会删除队尾或队列中间的节点，会导致索引越位。

## LinkedBlockingQueue

LinkedBlockingQueue（以下简称LBQ）底层由链表实现，初始化时若不显式设定，LBQ是近乎无界的，上限是2^11-1，若添加速度远大于删除速度，可能会导致内存溢出。值得注意的是，LBQ中维护了两把锁，添加和删除可以同时进行，因此，LBQ的吞吐速度远大于ABQ。除了底层数据结构不同，在方法的逻辑上，两者还是很像的。

### 主要结构/变量

```java
public class LinkedBlockingQueue<E> extends AbstractQueue<E>
        implements BlockingQueue<E>, java.io.Serializable {

    /**
     * 节点类，用于存储数据
     */
    static class Node<E> {
        // Node存放的值
        E item;

        /**
         * One of:
         * - the real successor Node
         * - this Node, meaning the successor is head.next
         * - null, meaning there is no successor (this is the last node)
         */
        Node<E> next;

        Node(E x) { item = x; }
    }

    /** 阻塞队列的大小，默认为Integer.MAX_VALUE */
    private final int capacity;

    /** 当前阻塞队列中的元素个数 */
    private final AtomicInteger count = new AtomicInteger();

    /**
     * 阻塞队列的头结点
     */
    transient Node<E> head;

    /**
     * 阻塞队列的尾节点
     */
    private transient Node<E> last;

    /** 获取并移除元素时使用的锁*/
    private final ReentrantLock takeLock = new ReentrantLock();

    /** notEmpty条件对象，当队列没有数据时用于挂起执行删除的线程 */
    private final Condition notEmpty = takeLock.newCondition();

    /** 添加元素时使用的锁如 put, offer, etc */
    private final ReentrantLock putLock = new ReentrantLock();

    /** notFull条件对象，当队列数据已满时用于挂起执行添加的线程 */
    private final Condition notFull = putLock.newCondition();

}
```

在LBQ中，增删操作分别获取putLock、takeLock，同时，还具有**交叉唤醒机制**，比如存入元素后若原本元素个数为0，还会主动唤醒消费者队列中的线程。

<font color=red>疑问：为什么ABQ需要一把锁，增删不能同时进行；LBQ需要两把锁，增删可以同时进行？</font>

ABQ的底层数据结构是一个循环数组，有时候，增、删的操作可能操作的内存空间是同一部分，不仅是同样操作count，而count不保证可见性，修改不采用CAS。而且putIndex、takeIndex可能重合（空、满）、即使先判断count，由于操作不具有原子性，CPU的指令重排序可能导致两个线程同时操作一个槽位中的同一个值----数值被覆盖。

LBQ的底层数据结构为链表，且不循环。增操作的是last.next，删操作的是head.item，就算head、last指向同一个节点，也操作的是不同的内存空间，不会重合，而同一时间同样操作的线程只有一个能持锁，所以绝对不会出现安全问题。即使同样存在重排序等问题，但LBQ并不依赖count进行判断是否可操作，count只用于唤醒和睡眠。

### 主要方法

LBQ的方法、接口和ABQ是相同的，只是实现不同，同样包含插入方法三个、删除方法三个

#### 插入

offer方法（）

```java
public boolean offer(E e) {
    	// 若插入null节点直接报错
        if (e == null) throw new NullPointerException();
        final AtomicInteger count = this.count;
    	// 队列已满，直接退出
        if (count.get() == capacity)
            return false;
        final int c;
        final Node<E> node = new Node<E>(e);
        final ReentrantLock putLock = this.putLock;
    
    	// 取takeLock锁
        putLock.lock();
        try {
            // 再次判断，在这一段时间里，是否队列变为已满状态
            if (count.get() == capacity)
                return false;
            // 入队
            enqueue(node);
            // 获取count
            c = count.getAndIncrement();
            // 若自增后的count未达最大值
            if (c + 1 < capacity)
                // 唤醒下一个put线程
                notFull.signal();
        } finally {
            putLock.unlock();
        }
    	// 若未自增时，count为0，即原本队列为空，现在队列至少有一个值了，唤醒take节点
        if (c == 0)
            signalNotEmpty();
        return true;
}
```

<font color='red'>`count.get()`和`count.getAndIncrement()`的区别是什么？</font>

count是原子类，get直接获取count的值，getAndIncrement方法进行的操作：获取count的值，返回count值，count++。

<font color=red>前面判断`if (count.get() == capacity)`后，为什么直接`return false`，而后面`if(c == 0)`时，又signalNotEmpty()唤醒take线程？</font>

if(count.get() == capacity)成立，说明此时队列容量已达最大，既然容量已满，那么肯定有take线程被唤醒在进行操作，其他take线程在Condition中等待，正在take的线程完成后会自动判断是否唤醒后续线程，所以此时offer线程不需要唤醒线程。

而后面c == 0，说明在这个线程进行入队之前，队列都是空的，判断c == 0时已经释放锁，可能会有其他线程进入进行入队，那么此时count从0变为大于等于1，需要唤醒消费者线程进行take。

<font color=red>`signalNotEmpty()`方法源代码如下，为什么唤醒消费者线程需要持有`takeLock`？</font>

```java
private void signalNotEmpty() {
    final ReentrantLock takeLock = this.takeLock;
    takeLock.lock();
    try {
        notEmpty.signal();
    } finally {
        takeLock.unlock();
    }
}
```

**唤醒消费者线程**和**消费者线程进行出队**都需要修改notEmpty条件队列，唤醒线程也需要加锁是为了同时只有一个线程能修改条件队列，保证线程安全。

put方法（阻塞入队）

```java
public void put(E e) throws InterruptedException {
    	// 若存入为null，直接抛出异常
        if (e == null) throw new NullPointerException();
        final int c;
        final Node<E> node = new Node<E>(e);
        final ReentrantLock putLock = this.putLock;
        final AtomicInteger count = this.count;
        putLock.lockInterruptibly();
        try {
            // 与offer类似，但put中若队列已满会进入Condition挂起，等待唤醒
            while (count.get() == capacity) {
                notFull.await();
            }
            enqueue(node);
            c = count.getAndIncrement();
            if (c + 1 < capacity)
                notFull.signal();
        } finally {
            putLock.unlock();
        }
        if (c == 0)
            signalNotEmpty();
    }
```

put方法几乎和offer一模一样，只是将队列已满就退出发，返回false改为了进入Condition。

#### 删除/出队

remove

```java
public boolean remove(Object o) {
    if (o == null) return false;
    // 同时获取两把锁，因为remove会影响整个队列。
    fullyLock();
    try {
        for (Node<E> pred = head, p = pred.next;
             p != null;
             pred = p, p = p.next) {
            if (o.equals(p.item)) {
                // 删除节点
                unlink(p, pred);
                return true;
            }
        }
        return false;
    } finally {
        fullyUnlock();
    }
}

void unlink(Node<E> p, Node<E> pred) {

    p.item = null;
    pred.next = p.next;
    if (last == p)
        last = pred;
    if (count.getAndDecrement() == capacity)
        notFull.signal();
}
```

poll（非阻塞出队，若无元素直接返回null）

```java
public E poll() {
         //获取当前队列的大小
        final AtomicInteger count = this.count;
        if (count.get() == 0)//如果没有元素直接返回null
            return null;
        E x = null;
        int c = -1;
        final ReentrantLock takeLock = this.takeLock;
        takeLock.lock();
        try {
            //判断队列是否有数据
            if (count.get() > 0) {
                //如果有，直接删除并获取该元素值
                x = dequeue();
                c = count.getAndDecrement();
                // c>1，说明未take之前至少有两个元素，取走一个后，队列未空，继续唤醒等待在条件对象notEmpty上的消费线程
                if (c > 1)
                    notEmpty.signal();
            }
        } finally {
            takeLock.unlock();
        }
        // c == capacity，说明原本队列是满的，现在取走一个，则可以唤醒生产线程
        if (c == capacity)
            signalNotFull();
        return x;
    }

  private E dequeue() {
        Node<E> h = head;//获取头结点
        Node<E> first = h.next; 获取头结的下一个节点（要删除的节点）
        h.next = h; // help GC//自己next指向自己，即被删除
        head = first;//更新头结点
        E x = first.item;//获取删除节点的值
        first.item = null;//清空数据，因为first变成头结点是不能带数据的，这样也就删除队列的带数据的第一个节点
        return x;
    }

```

take方法（阻塞出队）

```java
public E take() throws InterruptedException {
        final E x;
        final int c;
        final AtomicInteger count = this.count;
        final ReentrantLock takeLock = this.takeLock;
    	// 可打断式加锁
        takeLock.lockInterruptibly();
        try {
            // 自旋，若队列为空便挂起
            while (count.get() == 0) {
                notEmpty.await();
            }
            x = dequeue();
            c = count.getAndDecrement();
            // 队列还有元素，唤醒下一个消费线程
            if (c > 1)
                notEmpty.signal();
        } finally {
            takeLock.unlock();
        }
    	// 原本队列是满的，于是唤醒生产线程
        if (c == capacity)
            signalNotFull();
        return x;
}
```

### 交叉唤醒机制

在LinkedBlockingQueue中，不管是入队还是出队，都会检查**修改前后的count**是否满足不同条件，会尝试唤醒两个条件队列中的线程。

<font color=red>以下的`c`都是`count`被修改前的值！</font>

- 入队（生产者线程）：
  - 若修改后，`c + 1 < capacity`，即修改后依然有空位，继续唤醒生产者线程`signalNotFull()`
  - 若修改前，`c == 0`，即原本队列为空，唤醒消费者线程`signalNotEmpty()`
- 出队（消费者线程）：
  - 若修改后，`c > 1`（等价**`c - 1 > 0`**），即修改后至少还有一个元素，继续唤醒消费者线程`signalNotEmpty()`
  - 若修改前，`c == capacity`，即原本队列为满，唤醒生产者线程`signalNotFull()`

## 阻塞队列总结

阻塞队列是能够进行阻塞入队、出队的队列，常用于消费者-生产者模型。主要的实现有基于循环数组的`ArrayBlockingList`和`LinkedBlockingList`，提供接口相同，但内部实现不同。

- ArrayBlockingList内部维护一把锁、两个条件队列，同时只能有一个线程获取可重入锁，进行入队或出队。

- LinkedBlockingList内部维护两把锁、两个条件队列，同时可以有一个消费者、一个生产者线程，分别持有takeLock、putLock，分别进行出队、入队，互不干涉

两个阻塞队列的接口

| 不同情况                 | 入队                              | 出队                                  |
| ------------------------ | --------------------------------- | ------------------------------------- |
| 返回布尔值，失败抛出异常 | `add(E e)`     成功true，失败异常 | /                                     |
| 返回布尔值，不抛出异常   | `offer(E e)`  成功true，失败false | `poll()`  成功返回元素，失败返回false |
| 阻塞入队、出队           | `put(E e)`     成功void，失败阻塞 | `take()`  成功返回元素，失败阻塞      |
